# Swarm State — Issue #266: Collision Handle-Space Scoring Regression

## Last updated
2026-06-25T15:00:00Z

## Completed
- Root-cause analysis:
  - `PhysicsSystem.step()` drains Rapier collision events as **collider handles**.
  - `rebuildHandleCaches()` stores **rigid-body handles** in `bumperHandleSet`, `targetHandleSet`, `ballHandleSet`, etc.
  - Bumpers have two colliders per body (solid ball + sensor cylinder), so collider and body handle spaces diverge immediately.
  - The dispatcher was comparing collider handles against body-handle sets, so valid bumper/ball collisions never matched, producing zero scores.
- Added debug-flag-gated instrumentation counters in `src/game/game-physics-controller.ts`:
  - `rawCollisionEvents`, `knownObstacleMatches`, `bumperMatches`, `awardScoreCalls`
  - Surfaced in `src/game-elements/debug-hud.ts` and `src/game.ts` debug HUD update path.
- Implemented Phase 1 fix:
  - Convert collider handles to parent **body handles** at the event boundary in:
    - `stepPhysics` fixed-fixed pre-filter.
    - `processCollision`.
    - `processContactForce`.
  - All subsequent portal/death/lookup/dispatch checks now use body-handle space consistently.
- Added regression test:
  - `tests/collision-handle-space-scoring.test.ts` (Vitest)
  - Supplies collider handles whose parent bodies live in `bumperHandleSet`/`ballHandleSet` and asserts a non-zero score.
  - Includes a guard proving that body-handle-only events do not score.
- Verified:
  - `npx tsc -b` ✅ clean
  - `npm run lint` ✅ 0 errors (3 pre-existing warnings in unrelated files)
  - `npm test` ✅ 421 passed (39 test files)
  - `npm run build` ✅ (including WASM build)
  - Browser verification via Playwright MCP: teleporting the ball onto the center bumper produces `score=200`, `bumperMatches=2`, `awardScoreCalls=2`.
