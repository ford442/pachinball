## Adventure Track-Switch Hardening — 2026-06-05

### Issues identified
1. **Re-entrant portal deactivation:** `AdventureMode.updateExitPortal()` emits `PORTAL_ENTERED` then immediately calls `deactivateExitPortal()`. The event handler chains into `slotAdventure.switchToTrack()` → `adventureMode.switchToTrack()` → `deactivateExitPortal()` again. After unwinding, `getPortalSensorHandle()` returns `-1` and `physicsController.unregisterPortalSensor(-1)` is invoked in the `PORTAL_ENTERED` handler.
2. **Potential physics body leak in `clearTrack()`:** `kinematicBindings` and `animatedObstacles` arrays were zeroed (`[]`) without ensuring their underlying Rapier bodies are also removed from the world. Current track builders happen to push these bodies to BOTH `adventureBodies` AND the specialty arrays, making the leak theoretical, but the pattern is fragile.
3. **Missing integration test:** No test exercised the full `PORTAL_ENTERED` → `onPortalEntered` → `advanceToNextTrack` → `switchToTrack` chain.

### Fixes applied
1. **`src/game/game-systems-init.ts`** — `PORTAL_ENTERED` handler now guards `unregisterPortalSensor` with `enteredHandle !== -1`. Added a comment explaining the re-entrancy: when `onTrackAdvanced` chains into `switchToTrack()`, `deactivateExitPortal()` already emits `PORTAL_DEACTIVATED` which unregisters the valid handle; the only case where we still need to unregister in `PORTAL_ENTERED` is end-of-campaign (`nextTrackId === null`).
2. **`src/adventure/adventure-mode.ts`** — `clearTrack()` now defensively removes bodies from `kinematicBindings` and `animatedObstacles` explicitly before zeroing those arrays, future-proofing against track builders that might forget to add bodies to `adventureBodies`.
3. **`tests/adventure-track-switch.test.ts`** — Added three new tests:
   - *defensively removes orphan bodies from kinematicBindings and animatedObstacles* — injects a body that is NOT in `adventureBodies`, verifies `switchToTrack()` still cleans it up.
   - *onPortalEntered chains through onTrackAdvanced into switchToTrack* — full integration test using real `AdventureProgressionSupervisor` + `AdventureTrackProgression`, simulates score-goal resolution, portal entry, and asserts `GameSlotAdventure.switchToTrack()` is called with `CYBER_CORE`.
   - *does not call switchToTrack when campaign ends* — completes every track in the catalog, enters portal, asserts `switchToTrack` is skipped because `getNextTrackId()` returns `null`.
