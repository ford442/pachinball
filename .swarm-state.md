#### 3. Track-switch path
**Trace:** portal entry → `AdventureMode.updateExitPortal()` teleports ball & emits `PORTAL_ENTERED` → `GameSystemsInitializer` handles `PORTAL_ENTERED` → calls `AdventureProgressionSupervisor.onPortalEntered()` → `advanceToNextTrack()` → `callbacks.onTrackAdvanced?.(nextTrackId)`.

**Gap:** `onTrackAdvanced` callback is **not wired** in `GameSystemsInitializer`. The `PORTAL_ENTERED` handler updates progression state (`setCurrentTrack`, `initializeTrack`, `startTrack`) but **never tears down the old track geometry/physics** and **never builds the new track**. `AdventureMode.start()` returns early if `adventureActive` is already true, so calling it without `end()` first is a no-op.

**Fix status:** Gap is in orchestrator (`src/game/game-systems-init.ts` / `src/game/game-slot-adventure.ts`), outside permitted file list. Documented here for follow-up.

#### 4. Obstacle EventBus subscriptions
- `AdventureCinematicTriggers` — has `clearEventBus()` + `dispose()`. ✓
- `AdventureGoalSystem` — has `clearEventBus()` + `dispose()`. ✓
- `AdventureGoalTracker` — has `clearEventBus()` + `dispose()`. ✓
- `AdventureUIStateManager` — has `clearEventBus()` + `dispose()`. ✓

**Gap:** `dispose()` / `clearEventBus()` is only called on full game teardown (`GameDisposer`), not during intra-session track switches. Since these systems subscribe to singleton lifecycle events (`adventure:start`, `game:over`), subscriptions correctly persist across tracks. However:

- `AdventureUIStateManager.reset()` is **not called** during track switch, leaving stale goal elements.

**Fix status:** Orchestrator gap; documented for follow-up.

## Adventure Track-Switch Teardown + Rebuild — 2026-05-29 (Iteration 3)

### Objective
Wire the missing orchestration so that a portal jump from track A → track B actually tears down old geometry/physics, builds the new track, fires the track-start cinematic, and resets the goal UI.

### Approach
Chose to put the heavy lifting in `game-slot-adventure.ts` and keep `game-systems-init.ts` thin:
1. `AdventureProgressionSupervisor.onPortalEntered()` reordered so `reset()` happens **before** `advanceToNextTrack()`. This makes it safe for the `onTrackAdvanced` callback to call `startTrack()` without having it overwritten.
2. `AdventureMode.switchToTrack(newZone)` added — tears down old track via extracted `clearTrack()`, builds new track, updates camera preset, and emits `ZONE_ENTER`. Does NOT touch `followCamera` or `adventureActive` state.
3. `GameSlotAdventure.switchToTrack(trackId)` added — validates track ID, calls `adventureMode.switchToTrack()`, then drives `adventureCinematicTriggers.onTrackStart()`, `adventureUIStateManager.reset()`, `adventureGoalTracker.initializeTrack()`, `adventureProgressionSupervisor.startTrack()`, and display updates.
4. `GameSystemsInitializer` wired `onTrackAdvanced` in `AdventureProgressionSupervisor` construction to delegate to `slotAdventure.switchToTrack()`.
5. Removed the redundant `setCurrentTrack` / `initializeTrack` / `startTrack` / `setTrackInfo` block from the `PORTAL_ENTERED` handler in `GameSystemsInitializer` since the callback now handles it.
