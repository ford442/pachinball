# Iteration 2 — FEVER Trigger Identification

## Searched `src/game.ts` for fever-related logic

- `powerupActive` / `powerupTimer`: declared and reset, **never set to `true`**.
- `comboCount` / `comboTimer`: incremented on every bumper hit, reset when timer expires or ball is lost.
- `effects?.currentLightingMode === 'fever'`: checked in update loop (line 2783) but **never set to `'fever'`**.
- `BALL_SPAWN_CONFIG.feverMultipliers`: configured but **never referenced in gameplay code**.

## Conclusion: best trigger condition

**`comboCount >= 10` on a bumper hit**.

Rationale:
- Represents a sustained rapid bumper chain (10 consecutive hits within 1.5s of each other).
- Maps naturally to the existing combo/multiplier system.
- `powerupActive` is dead code; reviving it would require inventing a new activation event (e.g., item pickup) that doesn't exist.
- The `EffectsSystem` already supports `lightingMode: 'fever'` and fever trails; they just need to be triggered.

FEVER should end when `comboCount` resets to 0 (combo timer expiry or ball loss).

# Iteration 3 — Implementation

## Changes made

### 1. `src/game/game-state.ts`
- Added `setDisplaySystem(display: DisplaySystem | null): void` method to properly wire the display system after it is instantiated.
- Existing `handleStateTransition()` already drives `DisplayState.IDLE` for `MENU` and `GAME_OVER` — no changes needed there.

### 2. `src/game.ts`
- **Fixed init order**: after `new DisplaySystem(...)` (~line 1197), added `this.stateManager.setDisplaySystem(this.display)` so `GameStateManager` actually has a reference to the display system.
- **FEVER trigger**: in bumper collision handler (~line 2887), after `comboCount++`, added:
  ```typescript
  if (this.comboCount >= 10 && this.display?.getDisplayState() === DisplayState.IDLE) {
    this.display?.setDisplayState(DisplayState.FEVER)
    this.effects?.setLightingMode('fever', 0)
  }
  ```
- **FEVER exit on combo expiry**: in `updateCombo()` (~line 3150), when `comboTimer` expires:
  ```typescript
  if (this.display?.getDisplayState() === DisplayState.FEVER) {
    this.display?.setDisplayState(DisplayState.IDLE)
    this.effects?.setLightingMode('normal', 0)
  }
  ```
- **FEVER exit on ball loss**: in `handleBallLoss()` (~line 2992), after resetting combo:
  ```typescript
  if (this.display?.getDisplayState() === DisplayState.FEVER) {
    this.display?.setDisplayState(DisplayState.IDLE)
    this.effects?.setLightingMode('normal', 0)
  }
  ```

## What was changed

## Verification of all 4 display layers
`DisplaySystem.setDisplayState()` is the single entry point that configures all layers for every state:
- **IDLE**: `defaultMedia` → shader grid + reels (video/image from `GameConfig.backbox` if configured)
- **REACH**: `stateMedia[REACH]` → shader speed 5.0, color `#ff0055`, reels visible
- **FEVER**: `stateMedia[FEVER]` → shader speed 10.0, color `#ffd700`, reels visible
- **ADVENTURE**: `stateMedia[ADVENTURE]` → shader speed 1.0, color `#00aa00`, reels hidden
- **JACKPOT**: `stateMedia[JACKPOT]` → shader speed 20.0, color `#ff00ff`, reels hidden

Video and image layers are toggled based on `media.videoPath` and `media.imagePath` for each state. With the init-order fix, `MENU` and `GAME_OVER` now correctly revert to `IDLE` through `GameStateManager` instead of relying on manual calls.

## Open questions for the developer
1. **FEVER combo threshold**: Is `comboCount >= 10` appropriate, or should it be higher/lower?
2. **FEVER vs JACKPOT/REACH priority**: Current logic only activates FEVER when display is `IDLE`, so it won't override a Jackpot or Reach animation. Is this the desired priority?
3. **PAUSED display behavior**: When the game is paused, should the display remain in its current state (e.g., FEVER) or revert to IDLE? Currently no change.
