## B. GitHub issue — decoupled Copilot lane (slot-reel WGSL / canvas parity)

> **Title:** Slot-reel render: confirm WGSL (WebGPU) path matches the canvas-texture fallback in `display-reels.ts`

**Context / motivation**
The backbox-display system is an active focus area. The slot mini-game (#261) renders its reels through `src/display/display-reels.ts`, while the primary shader layer (`src/display/display-shader.ts`) authors WGSL for WebGPU with a canvas fallback. It's unverified whether the slot reels take a true WGSL/WebGPU path or silently always use the canvas texture — and whether the two paths are visually equivalent (symbol crispness, spin blur, stop-snap). This is pure display work with **zero overlap** with the concurrent C++/WASM physics effort.

**Proposed approach (first pass — expand this)**
- Audit `display-reels.ts` for how it chooses WebGPU vs canvas (`engine.isWebGPU` check, per CLAUDE.md convention).
- Compare against `display-shader.ts`'s WGSL authoring + fallback structure; identify where the reel path diverges or stubs the shader route.
- If the WGSL path is missing/stubbed, implement it so reels render via a `ShaderMaterial` on WebGPU with the existing canvas texture as the documented fallback.
- Ensure symbol sampling, motion blur during spin, and spring stop-snap look equivalent on both paths.

**Acceptance criteria (rough)**
- `display-reels.ts` has an explicit, documented WebGPU-vs-canvas branch gated on `engine.isWebGPU`.
- The WGSL path renders reels without console/WASM/WebGPU errors; the canvas path is unchanged.
- A Vitest (or Playwright visual) check asserts both paths activate under the right engine and produce reels (no blank/black surface).
- `npx tsc -b`, `npm run lint`, `npm run build`, `npx vitest run` all pass.

**Boundaries**
Touch `src/display/display-reels.ts`, `src/display/display-shader.ts` (reference only if needed), and a test. **Do not** touch `native/**`, `src/wasm/**`, `src/game-elements/physics.ts`, or `src/game/game-physics-controller.ts` — those are an active parallel lane.

**Open questions for Noah**
- Is the WGSL reel path a real requirement, or is canvas-only acceptable for reels and we just document that decision?
- Should this reuse the CRT/scanline post-chain, or render reels pre-post?
- Playwright visual-regression snapshot, or a lighter unit assertion?
